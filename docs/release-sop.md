# 发布流程规范

## 文档目标

本文档用于说明项目从 `staging` 晋升到 `production` 的标准发布流程，只记录稳定流程和回滚原则。

## 一、发布总原则

1. 先验证 `staging`，再发布 `production`
2. 数据库变更必须可追溯、可回放
3. 代码发布与数据库变更都必须可定位到明确版本
4. 发布与回滚都要有责任人

## 二、发版前准备

发版前至少确认以下事项：

- 目标版本对应明确 git commit
- 本次数据库变更已进入 `sql/migrations/`
- `staging` 与 `production` 环境变量键名一致
- 两个环境的 Node.js 大版本一致
- 回滚责任人与发布窗口已确认

## 三、Staging 发布

标准步骤如下：

1. 拉取目标版本代码
2. 执行 `npm ci`
3. 执行 `npm run migrate:staging:dry-run`
4. 执行 `npm run migrate:staging`
5. 启动或重启 `staging` 服务
6. 验证 `/health`
7. 验证登录、权限和核心业务流程
8. 检查应用日志与数据库连接状态

## 四、Production 发布

只有在 `staging` 验证通过后，才进入正式发布。

标准步骤如下：

1. 备份正式数据库
2. 拉取与 `staging` 相同的目标版本代码
3. 执行 `npm ci`
4. 执行 `npm run migrate:prod:dry-run`
5. 执行 `npm run migrate:prod`
6. 启动或重启正式服务
7. 执行正式环境冒烟验证

## 五、回滚原则

### 1. 代码与数据分离

- 代码回滚与数据库回滚分开设计
- 不把两者混成一个临时动作处理

### 2. SQL 风险控制

- 涉及破坏性 SQL 时，必须提前准备回滚方案
- 对不可逆 migration，优先采用前滚修复

### 3. 回滚后的验证

无论执行代码回滚还是前滚修复，回滚后都必须重新验证：

- `/health`
- 登录流程
- 核心业务接口

## 六、发布文档约定

后续与发布相关的文档只保留以下内容：

- 稳定的发布顺序
- 环境切换规则
- 回滚原则
- 检查项标准

不记录以下内容：

- 某次发版的逐步过程
- 某次发布时执行过的临时命令
- 某次排障的详细流水
