# 项目管理模块开发规范

## 文档目标

本文档用于沉淀项目管理模块的长期开发规范，只记录原则性内容、规则性内容和统一约定，不记录阶段过程、临时排障过程或逐步实施流水。

## 一、模块边界

项目管理模块当前边界包括：

- 项目管理
- 项目成员管理
- 需求管理
- Bug 管理
- 项目统计
- 操作日志

不在当前模块边界内的内容包括：

- 复杂工作流引擎
- 甘特图
- 自动通知中心
- 文件附件中心
- 多级审批流

如后续新增功能，优先以独立子模块方式扩展，不直接破坏现有项目、需求、Bug、统计四类核心模型。

## 二、后端结构约定

项目管理模块后端按以下分层组织：

- `models/`：只负责数据访问与聚合查询
- `controllers/`：只负责参数校验、流程编排、响应结构
- `routes/`：只负责路由注册、鉴权中间件挂载
- `sql/migrations/`：只负责结构变更和必要的基线数据

新增能力时遵循以下规则：

1. 新增一个核心领域对象，优先补齐对应 `model + controller + route`
2. 列表查询和详情查询分开处理，不把所有查询逻辑堆到一个接口
3. 状态流转接口与普通编辑接口分离，避免状态变更被普通更新误伤
4. 工时更新、负责人更新等高频动作优先拆为独立接口

## 三、数据模型规则

### 1. 删除策略

项目、需求、Bug 统一采用软删除：

- 字段：`is_deleted`
- 查询默认过滤 `is_deleted = 0`
- 统计只统计未删除数据

### 2. 审计字段

以下字段在新增表时默认保留：

- `created_at`
- `updated_at`
- `created_by`
- `updated_by`

### 3. 关联规则

- 需求必须归属项目
- Bug 可以只关联项目，也可以同时关联需求
- 若 Bug 关联需求，则该需求必须属于同一项目
- 项目成员表只保存项目内角色，不替代全局 RBAC

## 四、状态与流程规则

### 1. 项目状态

- `IN_PROGRESS`：进行中
- `COMPLETED`：已完成

### 2. 需求状态

- `TODO`
- `IN_PROGRESS`
- `DONE`

允许的最小状态流转：

- `TODO -> IN_PROGRESS`
- `IN_PROGRESS -> DONE`
- `IN_PROGRESS -> TODO`

### 3. Bug 状态

- `OPEN`
- `FIXING`
- `VERIFIED`
- `CLOSED`

允许的最小状态流转：

- `OPEN -> FIXING`
- `FIXING -> VERIFIED`
- `VERIFIED -> CLOSED`
- `VERIFIED -> FIXING`

### 4. 阶段字段

需求和 Bug 的阶段统一使用基础研发流程：

- `REQUIREMENT`
- `DEVELOPMENT`
- `TEST`
- `RELEASE`

阶段字段用于表达当前所处流程环节，状态字段用于表达任务完成度，两者不要混用。

## 五、权限规则

项目管理模块复用现有登录与 RBAC 体系，不单独建设认证系统。

当前权限设计原则：

- 查询类能力使用 `*.view`
- 新增类能力使用 `*.create`
- 编辑类能力使用 `*.edit`
- 删除类能力使用 `*.delete`
- 状态流转类能力使用 `*.transition`
- 成员维护类能力使用 `project.member.manage`
- 统计查看类能力使用 `project.stats.view`

新增功能时，优先复用现有权限前缀，避免出现重复语义的权限码。

## 六、业务线数据隔离规则

项目管理模块默认启用“按业务线隔离”的数据范围控制。

规则如下：

- 超级管理员可查看全部业务线数据
- 非超级管理员必须绑定唯一业务线
- 非超级管理员仅可访问绑定业务线下的数据
- 越权访问其他业务线时应返回 `403`

当前绑定关系通过 `pm_user_business_lines` 维护，约束为“每个用户唯一业务线”。
## 七、操作日志规则

项目管理模块的关键业务动作需要写入 `pm_activity_logs`。

建议至少记录以下动作：

- 创建
- 更新
- 删除
- 状态流转
- 阶段更新
- 指派负责人
- 工时更新
- 成员变更

日志记录要求：

- `entity_type` 必须明确为 `PROJECT`、`REQUIREMENT`、`BUG` 之一
- `entity_id` 必须可追溯到具体对象
- `action` 使用稳定的英文动作标识
- `action_detail` 使用便于运营和测试理解的中文描述
- `operator_user_id` 必须记录实际操作人

## 八、统计口径规则

### 1. 工时口径

- 预计工时和实际工时均按小时存储
- 人天统一按 `1 人天 = 8 小时` 计算
- 项目总工时 = 需求工时 + Bug 工时

### 2. 统计过滤规则

- 只统计 `is_deleted = 0` 的有效数据
- 成员统计以需求负责人和 Bug 负责人为统计来源
- 项目统计禁止直接多表硬连接后汇总，避免工时重复累加

### 3. SQL 设计原则

- 统计查询优先使用“分表先聚合，再统一汇总”的方式
- 复杂统计优先保证口径正确，再考虑 SQL 压缩
- 接口返回口径必须与页面展示口径保持一致

## 九、前后端协作规则

- 页面筛选项必须与后端枚举值保持一致
- 前端展示层如需使用“业务线”等业务化文案，应与后端 `project` 语义保持一一对应
- 前端显示文案可以中文化，但提交值必须使用约定枚举
- 列表页的高频操作优先采用弹窗或抽屉，不强依赖复杂跳转
- 状态和阶段展示统一使用标签，不同页面保持一致的颜色语义

## 十、文档记录规则

后续本模块文档统一遵循以下标准：

1. 只记录长期有效的架构、规则、约定、边界和口径
2. 不记录“第一批、第二批、第三批”这类阶段过程流水
3. 不记录临时排障步骤、测试步骤、手工验证明细
4. 只有当问题会沉淀为长期规则时，才写入文档
5. 新文档优先合并到现有主题文档中，避免碎片化新增

## 十一、推荐维护方式

当模块继续扩展时，优先更新以下文档，而不是新增大量过程文档：

- 架构变化：更新 `01-architecture.md`
- 表结构变化：更新 `02-database-design.md`
- 接口变化：更新 `03-backend-api-design.md`
- 页面结构变化：更新 `04-frontend-design.md`
- 开发规则变化：更新本文档
