# 项目管理模块测试环境规范

## 文档目标

本文档用于约束项目管理模块在本地测试环境 `staging` 下的开发、联调和迁移方式，只记录稳定规则，不记录单次迁移过程。

## 一、环境使用原则

项目管理模块后续默认在本地 `staging` 环境开发与验证。

统一原则如下：

1. 日常联调默认使用 `staging`
2. 测试通过后再考虑合并到正式环境
3. 未经确认，不直接在正式环境做结构和数据改动

## 二、当前环境约定

- 环境文件：`.env.staging`
- 默认服务端口：`3001`
- 默认数据库：`beixin_store_staging`
- 数据库变更目录：`sql/migrations/`

如环境信息后续调整，应直接更新本文档，不再新增迁移流水记录。

## 三、数据库变更规则

测试环境中的结构调整统一通过 migration 执行，不依赖手工建表。

执行原则：

1. 先执行 dry run
2. 确认 SQL 无误后再正式执行
3. migration 文件一旦进入团队协作，不随意改写历史版本

推荐顺序：

```powershell
npm run migrate:staging:dry-run
npm run migrate:staging
```

## 四、联调规则

### 1. 服务启动

本地联调默认以 `staging` 方式启动服务。

```powershell
npm run start:staging
```

### 2. 验证顺序

联调时建议按以下顺序检查：

1. `/health` 是否正常
2. 登录是否正常
3. 权限是否齐全
4. 项目、需求、Bug、统计接口是否返回正确
5. 前端页面是否与接口口径一致

### 3. 发布前顺序

统一遵循以下顺序：

1. 本地开发
2. 本地 `staging` 联调
3. 测试确认
4. 再合并到正式环境

## 五、测试账号规则

测试环境可以保留最小可用基线账号，但必须满足以下要求：

- 仅用于测试环境
- 不与正式环境混用
- 权限范围明确
- 如密码变更，应同步更新到团队内部安全记录中

当前测试环境已使用管理员账号作为联调基线，后续如有调整，直接维护本文档中的规则说明即可，不再记录单次创建过程。

## 六、数据边界规则

- 测试环境允许存在演示数据和联调数据
- 正式环境禁止直接写入未验证的测试数据
- 涉及账号、权限、基线角色等数据时，优先通过 migration 或初始化脚本维护
- 临时调试数据不作为文档长期内容保存

## 七、文档记录规则

与测试环境相关的文档后续只保留以下内容：

- 环境边界
- 迁移原则
- 联调规则
- 发布顺序
- 编码和数据安全约定

不再记录以下内容：

- 某次迁移做了哪些步骤
- 某次手工验证跑了哪些接口
- 某次创建了哪些临时测试数据
- 某次排障的完整流水
