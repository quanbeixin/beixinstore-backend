# 项目管理模块测试数据规范

## 文档目标

本文档用于沉淀测试数据的长期规则，特别是中文数据、演示数据和联调数据的写入规范，避免再次出现数据编码和脏数据问题。

## 一、测试数据原则

测试数据的目标是支撑联调、演示和验收，不是长期沉淀所有调试痕迹。

统一原则如下：

1. 测试数据应服务于页面展示和接口验证
2. 测试数据命名应可识别、可追踪
3. 测试数据应尽量保持真实业务语义
4. 临时数据完成用途后应清理

## 二、中文数据写入规则

为避免中文显示为 `?`，后续写入中文测试数据时遵循以下规则：

1. 优先使用 Node.js 脚本写入
2. 数据库连接字符集显式指定为 `utf8mb4`
3. 如需批量导入，优先从 UTF-8 编码文件读取
4. 尽量不要直接使用 PowerShell 内联中文对象发请求或拼 SQL

## 三、推荐写入方式

推荐优先级如下：

1. migration 或初始化脚本
2. Node.js 数据脚本
3. 后端接口批量导入

不推荐方式：

- 终端直接手写大量中文 JSON
- 终端直接拼接大量中文 SQL
- 无编码声明的临时脚本

## 四、测试数据命名规则

### 1. 项目编码

测试项目编码建议使用可识别前缀，例如：

- `PM-STAGE-001`
- `STAGING-DEMO-001`

### 2. 文本内容

项目名、需求标题、Bug 标题应尽量具备业务语义，不使用无意义占位词，例如：

- 推荐：`北新商城项目管理升级`
- 推荐：`项目总览卡片数据不刷新`
- 不推荐：`测试项目1`
- 不推荐：`标题A`

## 五、排查顺序规则

当页面出现中文乱码、问号或显示异常时，统一按以下顺序排查：

1. 先查数据库原始值
2. 再查接口返回值
3. 最后查前端显示逻辑

如果数据库原始值已经是 `?`，优先判定为数据写入问题，而不是前端渲染问题。

## 六、演示数据维护规则

- 演示数据可以长期保留，但要保证内容可读
- 演示数据应覆盖项目、需求、Bug、统计四类主要场景
- 演示数据中的状态、阶段、工时应符合系统口径
- 操作日志如保留演示数据，应保持文案完整可读

## 七、文档记录规则

后续与测试数据相关的文档只记录以下内容：

- 数据写入原则
- 编码要求
- 命名规则
- 排查顺序

不记录以下内容：

- 某一次具体修了哪些记录
- 某一次执行了哪些 SQL
- 某一次验证返回了什么结果
