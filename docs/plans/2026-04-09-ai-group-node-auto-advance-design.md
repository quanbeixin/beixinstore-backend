# AI群消息驱动自动过节点（半自动）落地方案

## 1. 目标与边界

### 1.1 业务目标
- 降低“需求已具备过节点条件但未及时流转”的人工延迟。
- 在不牺牲可审计性的前提下，利用群聊信号辅助节点推进。
- 首期采用“半自动”模式：AI给出建议，系统自动执行仅限高置信且证据完整场景。

### 1.2 已确认策略（当前版本）
- 模式：`B. 半自动`
- 自动执行阈值：`方案2`（置信度 `>= 0.80` 且证据完整）
- 未达阈值：生成“待确认建议”，由负责人一键确认过节点。

### 1.3 范围（首期）
- 数据源：需求绑定群（飞书）文本消息。
- 识别事件：开发完成、提测完成、测试完成、验收通过、阻塞解除（首期白名单）。
- 目标动作：触发“节点推进建议”与“可自动推进执行”。

### 1.4 暂不纳入（后续迭代）
- 语音/图片OCR作为主输入（首期仅做可扩展预留）。
- 跨群跨需求智能归因（首期依赖需求已绑定群）。
- 完全无人值守自动推进（需稳定期后再评估）。

## 2. 总体架构（可开发）

### 2.1 链路
1. 飞书群消息事件进入 `ingestion`。
2. 消息标准化（去噪、去引用、去机器人回声）。
3. AI结构化抽取：`阶段信号 + 证据项 + 置信度 + 反证`。
4. 规则引擎二次校验：
- 当前节点是否合法
- 是否满足“证据完整性规则”
- 是否命中反向信号（如“回滚”“阻塞中”）
5. 决策输出：
- `AUTO_ADVANCE`（自动推进）
- `SUGGEST_ONLY`（生成待确认建议）
- `REJECT`（忽略并记录原因）
6. 全链路审计：保存输入、抽取结果、决策原因、执行结果。

### 2.2 模块拆分
- `MessageIngestionService`：接收与落库群消息事件。
- `AiSignalExtractor`：调用LLM进行结构化抽取。
- `NodeAdvancePolicyEngine`：业务规则与阈值判断。
- `NodeAdvanceExecutor`：执行过节点与幂等控制。
- `NodeAdvanceAuditService`：审计、回放、追踪。

### 2.3 数据模型（新增）
- `ai_signal_events`
- `ai_node_advance_candidates`
- `ai_node_advance_actions`
- `ai_node_advance_feedback`

## 3. AI技术与工具选型（推荐）

### 3.1 必需AI能力
- 意图识别：从自然语言识别“阶段完成/阻塞/撤销”等业务语义。
- 结构化抽取：严格输出JSON，避免自由文本难以落库。
- 置信度估计：用于自动执行门槛判断。
- 证据抽取：提取“谁、何时、做了什么、关联对象”。
- 反证识别：识别“未完成、待验证、回退”等否定信号。

### 3.2 推荐工具组合（首期）
- LLM推理层：OpenAI `Responses API` + `Structured Outputs(JSON Schema)`。
- 编排层：服务内封装 `AiSignalExtractor`（先不引入重编排框架，降低复杂度）。
- 评测层：离线标注集 + 回放评测脚本（Precision/Recall/F1 + 误判成本权重）。
- 观测层：请求日志、模型版本、schema版本、trace_id全量记录。
- 风控层：关键词黑白名单 + 规则引擎兜底（AI结果必须过业务规则）。

### 3.3 为什么这样选
- 首期目标是“快上线+可控”，不是追求最复杂Agent架构。
- 结构化输出可直接对接策略引擎，减少解析脆弱性。
- 先做“AI抽取 + 规则决策”双层门控，可显著降低误推进风险。

## 5. 验收口径（Go/No-Go）

### 5.1 功能验收
- 从群消息可生成结构化候选，字段完整率 `>= 98%`。
- 满足阈值且证据完整时，自动推进成功率 `>= 95%`。
- 不满足阈值时，100%进入“待确认建议”，不可直接推进。

### 5.2 质量验收
- 误推进率（错误推进/总推进）`<= 1%`（灰度期目标）。
- 重复推进率为 `0`（幂等保障）。
- 核心链路可追溯率 `100%`（每次决策都有审计记录）。

### 5.3 运营验收
- 可按项目/需求开关功能。
- 支持快速停用（kill switch）。
- 提供每日监控看板：候选数、自动推进数、人工确认率、误判样例。

---

## 6. 风险与兜底

- 风险1：语义歧义导致误判。  
  兜底：提高阈值 + 反证优先 + 半自动人工确认兜底。

- 风险2：模型输出漂移。  
  兜底：模型版本固定、schema版本化、回放集回归。

- 风险3：业务状态机复杂导致越权推进。  
  兜底：执行前强校验“当前节点+权限+前置条件”，不满足即拒绝。
